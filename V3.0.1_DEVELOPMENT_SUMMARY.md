## Why This Approach

### Problem Recap
- **v2.0.0 Issue**: DigitalAudioMessageHandler hook modifies unused data (firmware response is ignored by Android)
- **Root Cause**: Hardware filters group calls **BEFORE** sending RECEIVE_START to Android
- **Hardware Bug**: contactType=2 (ALL mode) accepted by firmware but still filters group calls
  - Evidence: All test session UART logs show `08 00 02` (contactType=2) but group calls still blocked

### Solution Strategy: Find Firmware Bypass

Instead of fighting the broken contactType=2 implementation, discover undocumented commands that might:
- **Disable RX filtering entirely**
- **Force promiscuous mode** (accept all DMR traffic)
- **Override group validation**
- **Enable debug/diagnostic modes** with relaxed restrictions

## What Success Looks Like

### Positive Indicators in UART Logs

**1. Non-Error Response:**
```
TX: CMD=0x4D RW=1 SR=1 Body=01
RX: CMD=0x4D RW=0 SR=0 Body=[data]  ← Command accepted!
```

**2. Large Response Payloads:**
Commands that return >8 bytes of data likely perform complex operations.

**3. State Changes:**
Commands that alter subsequent behavior when sent.

### Testing During Live Group Calls

**Critical Test:**
1. Have someone call a group NOT in RX list
2. Send discovered commands via broadcast
3. Check if call suddenly becomes audible
4. **That command is the bypass!**

## Technical Details

### Broadcast Intent Protocol
```java
Intent intent = new Intent("com.dmrmod.SEND_DEBUG_PACKET");
intent.putExtra("CMD", 0x4D);    // Command code 0-255
intent.putExtra("RW", 1);         // 0=Response, 1=Query/Set
intent.putExtra("SR", 1);         // 0=Success, 1=Request
intent.putExtra("BODY", "0102");  // Optional hex payload
context.sendBroadcast(intent);
```

### UART Packet Structure
```
HEAD: 0x68
CMD:  0x00-0xFF  ← We're fuzzing this
RW:   0=Read response, 1=Write/Query
SR:   0=Success, 1=Set/Request, 2=Checksum error
LEN:  Body length
BODY: Variable payload
CRC:  Checksum
TAIL: 0x10
```
