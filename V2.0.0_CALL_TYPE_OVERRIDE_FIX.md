# DMRModHooks v2.0.0 - Call Type Override Fix

## 🎯 Problem Solved

**Root Cause Discovered via UART Analysis:**
The DMR003.UV4T.V022 firmware has a bug where it **always reports `callType=0` (PRIVATE)** in CMD 0x2B (QUERY_DIGITAL_AUDIO_RECEIVE_INFO) responses, regardless of whether the incoming call is actually a private call or group call.

### Evidence from UART Logs (uart_20260227_221344.txt):
```
Test A (Private Call - WORKED):
  CMD 0x2B Response Body: 00 05 D8 00 00 A7 05 00 00
  Byte 0 = 0x00 (PRIVATE/PERSON)
  Result: Audio played ✅

Tests B & C (Group Calls - FAILED):
  CMD 0x2B Response Body: 00 05 D8 00 00 A7 05 00 00
  Byte 0 = 0x00 (PRIVATE/PERSON) ⚠️ INCORRECT!
  Result: Android filtered call ❌
```

**Hardware was lying:** Both call types reported identical `callType=0`, causing Android to correctly filter group calls thinking they were invalid private calls.

---

## 🔧 Solution Implementation

### v2.0.0 Changes

**Hook Location:** `DigitalAudioMessageHandler.handle()` - **beforeHookedMethod**

**Strategy:** Intercept CMD 0x2B response packets BEFORE Android processes them and override the incorrect `callType` byte (byte 0) based on the user's current `contactType` channel configuration.

### Logic Flow:

```java
// Read current contactType from DmrManager
DmrManager dmrManager = DmrManager.getInstance();
ChannelData currentChannel = dmrManager.getCurrentChannel();
int contactType = currentChannel.getContactType();

// Override hardware's incorrect callType based on user setting
if (contactType == 2) {
    // MON button ON (ALL mode)
    body[0] = 2;  // Force accept ALL calls (private + group)
} else if (contactType == 1) {
    // GROUP mode
    body[0] = 1;  // Let Android check RX group list
} else {
    // PERSON mode (contactType=0)
    body[0] = 0;  // Keep PRIVATE calls only
}
```

---

## 📊 9-Byte Payload Structure (CMD 0x2B)

Decoded from UART analysis:

```
Offset  | Bytes     | Value (Hex) | Value (Dec) | Meaning
--------|-----------|-------------|-------------|------------------
0       | 00        | 0x00        | 0           | Call Type (0=PRIVATE, 1=GROUP, 2=ALL)
1-2     | 05 D8     | 0x05D8      | 1496        | Contact/Group ID (little-endian)
3-4     | 00 00     | 0x0000      | 0           | Unknown/Reserved
5-8     | A7 05 00 00 | 0x000005A7 | 1447        | Caller DMR ID (little-endian)
```

**Note:** Hardware always sent byte 0 = 0x00 for both private and group calls in testing.

## 🛠️ Technical Details

### Implementation Notes:

1. **Hook Timing:** Uses `beforeHookedMethod` instead of `afterHookedMethod` to modify data BEFORE Android's handler processes it

2. **Byte Array Modification:** Directly modifies `body[0]` in the CMD 0x2B response packet - changes persist through Android's call handling logic

3. **Thread Safety:** DmrManager.getInstance() is called in hook context, should be safe as it's Android's main radio processing thread

4. **Backward Compatibility:** If contactType read fails, hardware's original callType is preserved (no worse than before)

### Dependencies:
- Requires LSPosed/EdXposed
- Target app: `com.pri.prizeinterphone`
- Hardware: DMR003.UV4T.V022 firmware (or compatible)

## 🐛 Known Limitations

1. **Hardware Inconsistency:** One of the three test calls (Tests A/B/C) never sent RECEIVE_START event to Android - this suggests hardware filtering still occurs BEFORE contactType check in some cases

2. **RX Event Filtering:** The hardware may still pre-filter some calls before Android sees them, contactType override only affects calls that reach Android

3. **Firmware Update Risk:** If Ulefone updates DMR firmware, packet structure may change requiring re-analysis
