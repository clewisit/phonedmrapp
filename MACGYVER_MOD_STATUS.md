## ❌ Deployment Blocker: APK Signature Mismatch

### The Problem

Android Package Manager and Magisk both reject the rebuilt APK because it has a different cryptographic signature than the original system app.

### Signature Comparison

| Attribute | Original System APK | Rebuilt APK |
|-----------|---------------------|-------------|
| **File** | original-system.apk | MacDMRUlephone-v0.1.apk |
| **Size** | 7.76 MB (8,136,003 bytes) | 7.63 MB (7,997,795 bytes) |
| **MD5** | F9B95B920B567D897C273418487A36C0 | C096A839F5C0550A74B114366D87ABEC |
| **Signature Type** | Platform (manufacturer keys) | Debug (Gradle auto-generated) |
| **versionCode** | 33 | 35 |
| **versionName** | V1.0 | 2.0-MacDMR |

### Error When Installing

```
INSTALL_FAILED_UPDATE_INCOMPATIBLE: Existing package com.pri.prizeinterphone 
signatures do not match newer version; ignoring!
```

### Why This Matters

1. **Android Security**: System apps signed with platform keys cannot be replaced by apps with different signatures
2. **Magisk Security**: Magisk module system refuses to overlay APKs with mismatched signatures as a security feature
3. **Package Manager**: Prevents signature downgrade attacks by enforcing signature consistency

**Error 2**: Resource ordering (BLOCKING)
```

```

**Root Cause**: Adding new resource IDs to `ids.xml` and `public.xml` breaks AAPT's expectation that attr resources come first in resource table compilation.

**Technical Details**:
- AAPT expects resources in specific order: attr, drawable, layout, string, id, etc.
- Adding IDs between existing IDs shifts resource addresses
- `public.xml` defines fixed resource IDs (0x7f090186, 0x7f090187, etc.)
- Manual ID insertion causes resource table corruption

**Attempts to Fix**:
1. ❌ Modified `public.xml` with sequential IDs → "First type is not attr"
2. ❌ Copied to WSL native filesystem → Same error
3. ❌ Removed invalid drawables → Still resource ordering error
4. ❌ Fresh decompilation without public.xml changes → Need IDs for findViewById

**Status**: ⏸️ BLOCKED - Resource compilation requires deep understanding of AAPT internals

## 🔬 Technical Lessons Learned

### Gradle Build System
1. **Package attribute conflict**: AGP 7.4.2+ uses `namespace` in build.gradle, having `package=` in manifest causes duplicate definition
2. **R.java caching**: Gradle aggressively caches generated R.java, requiring nuclear clean for new resource IDs
3. **Resource ID generation**: Happens at compile time via AAPT2, stored in R.txt then R.jar

### APK Structure
1. **Binary resources**: All XML files are compiled to binary AXML format
2. **Resource table**: resources.arsc contains binary mapping of IDs to resources
3. **DEX files**: Java classes compiled to Dalvik bytecode (classes.dex, classes2.dex, etc.)
4. **Signature**: META-INF/ folder contains signing certificates and manifests

### apktool Limitations
1. **Resource ordering**: AAPT expects specific resource type ordering
2. **Public IDs**: Changing public.xml IDs breaks resource table structure
3. **Binary resources**: Some resources don't decompile cleanly (drawables with $ in name)
4. **Framework dependencies**: Requires framework-res.apk for proper decompilation

### Magisk Module Security
1. **Signature verification**: Magisk checks APK signatures before overlaying
2. **System app replacement**: Can replace system apps but only with matching signatures
3. **Module structure**: Mirrors /system partition structure
4. **Installation**: Requires reboot to activate overlay
